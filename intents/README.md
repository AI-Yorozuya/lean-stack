# intents — 先寫規則，再寫 code

這個資料夾放 **INTENT**：用人話把「業務規則」寫清楚，**寫在 code 之前**。
寫程式時照著 INTENT 生 —— 規則是真相來源，code 只是實作。

> 為什麼？因為 bug 多半不是「程式寫錯」，而是「一開始就沒講清楚規則」。
> 把狀態、誰能做什麼、什麼不准發生先寫下來，AI 跟人都不會各自亂猜。

## 一個 INTENT 在描述什麼？

一個 **狀態機 + 權限**：東西有哪些狀態、誰能在什麼條件下把它從一個狀態推到另一個狀態、
以及什麼是「絕對不能破」的鐵則。

## 簡化語法

每條規則一行：

```
From-State --(Who: Action)--> To-State [Guard] {鐵則}
```

- **From-State**：動作前的狀態（例：`草稿`）。
- **Who**：誰能做（角色，例：`記帳人`）。
- **Action**：做什麼（例：`送出`）。
- **To-State**：動作後的狀態（例：`待審`）。
- **[Guard]**：前提條件，不成立就不能做（例：`[金額>0]`）。可省略。
- **{鐵則}**：永遠要成立的不變量 / 紅線（例：`{借貸必須相等}`）。可省略，但通常最重要。

例：

```
草稿   --(記帳人: 送出)--> 待審  [金額>0]      {一張單必須至少一筆分錄}
待審   --(審核人: 核准)--> 已過帳            {借方總額 == 貸方總額}
待審   --(審核人: 退回)--> 草稿
已過帳 --(系統: 鎖定)--> 已鎖   [已過會計期]  {已鎖的單不可再改}
```

## 權限 5W（anchor concept）

每個 Action 想清楚這 5 個 W，權限規則就齊了：

- **Who** — 哪個角色 / 身分。
- **What** — 對什麼資源 / 哪些欄位。
- **When** — 在什麼狀態 / 時間 / 條件下（對應 Guard）。
- **Where** — 哪個範圍（自己的？整個組織的？某租戶的？）。
- **Why** — 這條規則背後的業務理由（寫一句，未來改規則時知道能不能動）。

## 怎麼用

1. 複製 `_TEMPLATE.md` → `intents/<功能名>.md`，把狀態機 + 5W + 鐵則填好。
2. 照 INTENT 生後端（app / router / schema / model）與前端頁（見根目錄 `CLAUDE.md` 的「INTENT-first 加功能」）。
3. 規則變了 → 先改 INTENT，再改 code。

> 注意：**記帳 / bookkeeping 的 INTENT 還沒寫** —— 那是下一個由人來寫的步驟。這裡先備好骨架與範本。
