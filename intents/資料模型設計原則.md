# 資料模型設計原則

> 這是**教學模板**——model 要教「對的形狀」。可以簡單，但必須是「真的東西」該有的骨架，不是玩具。
> 每份 INTENT 的「名詞」段照這幾條設計，並附一張 Mermaid `erDiagram`（純文字，你可以直接改成自己要的）。
> 訂單當範例（見 [訂單管理](訂單管理.md)）。

## 四條共同鐵則（設計任何 model 都照這個）

### 1. 目錄與明細分離，明細存「快照」

可販售的東西（`Product`：品名、牌價）放**目錄**，是「當前真相」。
訂單明細（`OrderItem`）在**下單當下把品名＋單價抄一份存下來**（快照），是「歷史事實」。

- 之後目錄改價、改名、甚至下架，**歷史訂單原封不動**。
- 算舊訂單金額**永遠不 join 現在的目錄價**——那會竄改歷史。

> 這是電商 / ERP model 最經典、也最常被 naive 寫錯的一條。少了「目錄」這個實體，就沒地方演這一課。

### 2. 衍生值一律算出來，不手填

`小計 = 數量 × 單價(快照)`、`總額 = Σ 小計`。
存進 DB 欄位只是查詢快取，**真相是「算的」**（寫在 model 的 `save()` / `recalc_*()`，不接受手填）。

> 為什麼算的要寫在後端而不是相信前端？前端可以算給人看（體驗），但承重牆砌在後端才擋得住改 request 的破壞。

### 3. 業務識別碼 ≠ 資料庫主鍵

給人和客戶對帳看的是**單號**（`order_no`，例：`OR-000123`），DB 自增 `id` 是內部的事。
兩個一定要分開——不然學員會拿自增主鍵當單號往外露（可被猜、洩漏量體）。

### 4. 停用不刪

下架 / 離職 / 作廢用 `is_active=False`（或狀態欄），**不 DELETE**。
被引用的資料用 `PROTECT` 擋刪。歷史要留得住。

> 呼應 [會員管理](會員管理.md) 的 `{停用 = 關閉，資料保留}`。

## 簡單 ≠ naive：有意識地 park

模板刻意不做這些（它們是「加法」，加了不改變上面的正確性）：
庫存扣減、金流串接、發票、稅、多幣別、折扣、部分退款。

差別就在上面四條有沒有做對——**做對了，薄的 model 也是「真的」；做錯了，再多欄位也是玩具。**
每份 INTENT 的「park」段就是在明講「這是選擇，不是漏掉」。
