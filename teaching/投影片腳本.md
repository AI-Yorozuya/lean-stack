### S04 · 構成與後端溝通：篩選是怎麼運作的 ★原理
- **目標**：看懂「篩選」這組積木怎麼**構成**、又怎麼**跟後端要資料**——前端只收條件、送請求、畫結果，過濾的真相在後端。
- **畫面**：工具列特寫（`screens/s04-filter.png`）＋ 純圖（下方 sequence）。
- **標註**：框「工具列 ＝ Input ＋ Select ＋ 搜尋 Button」與「表身下方 Pagination」；箭頭：Input→`q`、Select→`status`。
- **原理（資料怎麼流）**：
  1. 使用者打字（`q`）／選狀態（`status`）。
  2. 按搜尋 / Enter **才送**（不是每打一字就打 API）→ 前端把條件組成 params。
  3. `GET /member?q=…&status=…` → 後端 **query 過濾**（過濾的真相在這）。
  4. 後端回 `{ items, count }`。
  5. 前端 render 表格；分頁在前端切頁（資料量小、整包載入）。
- **一句話原理**：篩選＝「前端收條件、組 params、送請求、畫結果」；**怎麼濾在後端 query，不在前端**。

```mermaid
sequenceDiagram
    participant U as 使用者
    participant V as 前端 View
    participant A as api/member.js
    participant B as 後端 ninja
    U->>V: 打字 q / 選 status / 按搜尋
    V->>A: listMembers({ q, status })
    A->>B: GET /member?q=&status=
    B->>B: query 過濾（真相在這）
    B-->>A: { items, count }
    A-->>V: 回資料
    V->>V: render 表格 + 前端分頁
```
- **對應**：`MemberListView.vue` 的 `load()`、`src/api/member.js` 的 `listMembers`、`apps/member/apis.py` 的 `list_members`
